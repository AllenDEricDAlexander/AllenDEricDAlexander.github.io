Docker 深度入门：从环境安装、核心原理到生产部署实践

欢迎你来读这篇博客。

这篇文章主要聊 Docker：从环境安装、核心概念、镜像与容器、网络、数据卷、端口映射，到常见应用部署、容器转镜像、Dockerfile、多阶段构建、Compose
编排、日志排查、资源限制、备份恢复和生产建议。

Docker 本身不神秘，它解决的核心问题可以概括成一句话：

把应用和运行环境一起打包，让它在不同机器上尽可能一致地运行。

以前我们常说：“我本地能跑啊。”
Docker 出来之后，这句话的杀伤力下降了不少。当然，Docker 不是银弹，写不好 Dockerfile、卷挂错、端口乱映射、镜像不瘦身，照样能把环境搞成“赛博泥潭”。

序言：Docker 解决了什么问题？

传统部署里，一个 Java 服务要上线，通常要关心：

  • 服务器有没有 JDK？
  • JDK 版本是不是对？
  • 配置文件放哪里？
  • 端口有没有冲突？
  • MySQL、Redis、Nginx 有没有装？
  • 依赖库版本是否一致？
  • 测试环境和生产环境为什么表现不一样？

Docker 的思路是：
不要只交付代码，而是交付一个包含应用运行环境的镜像。

镜像里可以包含：

  • 操作系统基础环境
  • JDK / Node.js / Python / Nginx 等运行时
  • 应用程序包
  • 启动命令
  • 环境变量约定
  • 暴露端口
  • 健康检查逻辑

然后在任何支持 Docker 的机器上启动：

这就是 Docker 最朴素，也最强大的价值。

一、Docker 环境准备

1. Docker 的几个核心组件

Docker 不是一个单独的命令，而是一组工具和组件：

组件 说明
Docker Client 命令行客户端，也就是我们常用的 docker 命令
Docker Daemon Docker 服务端进程，负责创建镜像、运行容器、管理网络和卷
Docker Image 镜像，可以理解为应用运行环境模板
Docker Container 容器，镜像运行起来之后的实例
Docker Registry 镜像仓库，例如 Docker Hub、Harbor、阿里云镜像仓库
Docker Compose 多容器编排工具，用 YAML 描述一组服务

整体架构可以用下面这张图理解：

    B --> D[运行中容器 Containers]
    B --> E[网络 Networks]
    B --> F[数据卷 Volumes]
    B <-->|push / pull| G[Registry 镜像仓库]

5. host 网络

host 网络模式下，容器直接使用宿主机网络。

特点：

  • 性能损耗更少
  • 没有容器网络隔离
  • 端口直接占用宿主机端口
  • Linux 下常用，Docker Desktop 场景要注意差异

一般只有在明确需要时才使用 host 网络。

6. none 网络

none 网络表示容器没有网络。

适合一些完全不需要网络的离线任务。

7. overlay 网络

overlay 网络主要用于多主机容器通信，例如 Docker Swarm 场景。
如果只是单机 Docker 或本地开发，一般不需要上来就研究 overlay。

2. docker commit 的适用场景

适合：

  • 临时调试
  • 保存实验环境
  • 迁移某个临时容器状态
  • 快速验证镜像变化

不适合：

  • 正式生产镜像构建
  • 团队长期维护
  • CI/CD 自动构建
  • 可追溯发布流程

生产环境更推荐 Dockerfile。
因为 Dockerfile 可读、可审查、可版本管理、可重复构建。

3. docker commit 注意点

docker commit 保存的是容器文件系统变更。
如果数据写在 volume 中，commit 不会把 volume 数据打进镜像。

所以你不要期待：

就能把数据库完整数据一起保存下来。
数据库数据如果挂载在 volume 中，需要单独备份 volume 或使用数据库 dump。

6. 数据必须挂载 volume

数据库、上传文件、业务持久数据，必须挂载 volume 或外部存储。

7. 容器只跑一个主进程

一般一个容器只运行一个主要服务。
不要把 MySQL、Redis、Nginx、Java 应用全塞进一个容器。那不是容器化，那是把一台小服务器塞进盒子里，精致但痛苦。

8. 镜像构建要可复现

推荐：

  • Dockerfile 版本管理
  • 使用 CI/CD 构建镜像
  • 使用明确 tag
  • 构建后推送镜像仓库
  • 部署时只拉取指定版本镜像

总结

Docker 的核心不是“会背命令”，而是理解它的几个抽象：

  • 镜像：应用运行环境模板
  • 容器：镜像运行实例
  • Dockerfile：镜像构建说明书
  • Registry：镜像分发中心
  • Network：容器通信方式
  • Volume：数据持久化方式
  • Compose：多容器应用编排方式

真正落地时，要重点注意：

  1. 端口映射要明确，别把内部端口和宿主机端口搞混。
  2. 数据库必须挂载 volume，不然删容器就是删青春。
  3. 生产镜像尽量使用 Dockerfile 和 CI/CD 构建，不要依赖 docker commit。
  4. Dockerfile 要利用缓存、多阶段构建、非 root 用户和 .dockerignore。
  5. Compose 适合单机多服务部署，但复杂调度、弹性伸缩和大规模服务治理还是 Kubernetes 更合适。
  6. 日志、资源限制、健康检查、版本 tag、备份恢复，这些才是生产环境真正救命的东西。

Docker 入门很快，但写好 Docker 需要工程习惯。
容器不是魔法盒，它只是把环境问题标准化了；如果镜像、网络、卷、配置、日志都设计得乱，那只是把混乱打包得更精致。
