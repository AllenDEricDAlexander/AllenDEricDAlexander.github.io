设计模式：代理模式

欢迎你来读这篇博客，这篇博客主要是关于代理模式。
其中包括代理模式的核心思想、适用场景、静态代理、JDK 动态代理、CGLIB 代理、Spring AOP 代理、优缺点，以及 Java 后端开发中订单服务增强的完整案例。

序言

在 Java 后端开发中，我们经常会遇到这样的问题：

我不想直接访问某个对象，而是希望在访问它之前或之后做一些控制。

比如调用订单服务：

现在希望在真正创建订单之前或之后增加一些逻辑：

  • 权限校验；
  • 参数校验；
  • 日志记录；
  • 耗时统计；
  • 事务控制；
  • 缓存处理；
  • 远程调用封装；
  • 限流保护；
  • 异常转换；
  • 监控埋点。

最简单的方式是直接修改 OrderService。

但是这样做会让业务代码越来越臃肿。

订单服务本来只应该关心“怎么创建订单”，现在却开始关心日志、权限、事务、监控、缓存。最后一个类里塞满了各种横切逻辑，业务方法像穿了十层棉袄，跑都跑不动。

代理模式就是为了解决这类问题：

给目标对象提供一个代理对象，由代理对象控制对目标对象的访问，并可以在访问前后增加额外逻辑。

简单说：

代理模式就是不直接找本人办事，而是先找代理人。

代理人可以帮你做很多事：

  • 判断你有没有资格访问；
  • 记录你什么时候访问；
  • 控制访问频率；
  • 延迟创建真实对象；
  • 把本地调用转成远程调用；
  • 在调用前后加增强逻辑。

这也是 Spring AOP、MyBatis Mapper、RPC 框架、事务管理等技术背后非常重要的思想基础。

正文

chapter 1：什么是代理模式

代理模式，英文是 Proxy Pattern，属于结构型设计模式。

它的定义是：

为其他对象提供一种代理，以控制对这个对象的访问。

代理模式一般包含三个角色：

  1. Subject 抽象主题：定义目标对象和代理对象的共同接口。
  2. RealSubject 真实主题：真正执行业务逻辑的对象。
  3. Proxy 代理对象：持有真实主题对象，在调用真实对象前后增加控制逻辑。

结构如下：

客户端调用的是 Subject 接口。

它可以拿到真实对象，也可以拿到代理对象。

代理对象内部再调用真实对象。

chapter 2：代理模式的核心思想

代理模式的核心思想是：

客户端不直接访问目标对象，而是通过代理对象间接访问目标对象。

代理对象可以在中间做控制。

例如：

  • 调用前增强；
  • 调用后增强；
  • 异常处理；
  • 访问控制；
  • 延迟加载；
  • 远程转发。

代理模式不是替换目标对象，而是控制访问目标对象。

chapter 3：为什么需要代理模式

代理模式常见价值有几个。

1. 控制访问

例如权限代理。

调用真实对象前先判断当前用户是否有权限。

2. 增强功能

例如日志、监控、事务。

3. 延迟加载

真实对象创建成本很高，只有真正使用时才创建。

4. 远程调用

客户端以为调用的是本地对象，代理对象内部把调用转成网络请求。

这就是 RPC 的核心思想之一。

5. 屏蔽复杂细节

代理对象可以隐藏真实对象的创建、连接、重试、序列化、反序列化、网络通信等细节。

chapter 4：代理模式的常见类型

代理模式有很多分类，常见包括：

  1. 静态代理；
  2. 动态代理；
  3. 远程代理；
  4. 虚拟代理；
  5. 保护代理；
  6. 缓存代理；
  7. 智能引用代理。

在 Java 开发中，最常讨论的是：

  • 静态代理；
  • JDK 动态代理；
  • CGLIB 动态代理；
  • Spring AOP 代理。

下面逐个讲。

chapter 6：静态代理的优缺点

1. 优点

静态代理的优点是：

  • 简单直观；
  • 不需要反射；
  • 代码容易理解；
  • 编译期就能检查类型；
  • 适合少量接口增强。

2. 缺点

静态代理的问题也很明显：

  • 每个接口都要写代理类；
  • 方法多时代码重复；
  • 目标类多时代理类爆炸；
  • 不够灵活；
  • 不适合通用横切逻辑。

比如系统里有 100 个 Service 都要加日志。

如果用静态代理，就要写很多代理类。

这显然不现实。

这时就需要动态代理。

chapter 10：JDK 动态代理的优缺点

1. 优点

  • Java 原生支持；
  • 不需要手写代理类；
  • 可以统一增强多个方法；
  • 适合基于接口的编程；
  • Spring AOP 可以使用 JDK 动态代理。

2. 缺点

  • 目标类必须实现接口；
  • 只能代理接口方法；
  • 反射调用有一定性能成本；
  • 代码理解成本比静态代理高；
  • 对 final 方法无能为力，因为接口代理不涉及类方法覆盖。

如果你的类没有接口，JDK 动态代理就不适合。

这时可以使用 CGLIB 代理。

chapter 11：CGLIB 动态代理

CGLIB 是一种基于继承的动态代理方式。

它的特点是：

通过生成目标类的子类来创建代理对象。

也就是说，CGLIB 不要求目标类实现接口。

它会在运行时生成一个子类，并重写目标类的方法，在方法前后加入增强逻辑。

结构类似：

CGLIB 底层依赖字节码生成技术。

在 Spring AOP 中，如果目标对象没有实现接口，通常可以使用 CGLIB 方式创建代理。

chapter 13：CGLIB 代理的限制

CGLIB 虽然不要求接口，但它也有限制。

1. 不能代理 final 类

因为 CGLIB 通过继承生成子类。

如果类是 final，不能被继承。

这种类无法被 CGLIB 代理。

2. 不能代理 final 方法

因为 final 方法不能被子类重写。

CGLIB 无法拦截这个方法。

3. 对构造方法和类结构有一定要求

CGLIB 需要创建子类对象，构造方法、可见性等因素都可能影响代理生成。

4. 调试成本更高

运行时生成子类，堆栈信息和普通调用不完全一样。

chapter 14：JDK 动态代理和 CGLIB 代理对比

对比项 JDK 动态代理 CGLIB 动态代理

是否能代理类 不能，只代理接口 可以代理类

生成方式 JDK Proxy 字节码生成子类
常见场景 接口型 Service 无接口类
Spring AOP 是否支持 支持 支持

简单记：

有接口，可以用 JDK 动态代理。

没接口，可以用 CGLIB。

不过在 Spring Boot 项目中，我们通常不需要手动创建这些代理对象，Spring 会根据配置和目标对象情况帮我们处理。

chapter 24：代理模式和装饰器模式的区别

代理模式和装饰器模式很像。

它们通常都：

  • 实现同一个接口；
  • 持有一个目标对象；
  • 在目标对象调用前后做一些事情。

但目的不同。

对比项 代理模式 装饰器模式
核心目的 控制访问 动态增强功能
关注点 是否允许访问、如何访问 给对象叠加额外职责
目标对象 通常由代理控制 通常由客户端传入并自由组合
使用场景 权限、事务、远程代理、懒加载 日志、重试、限流、缓存、缓冲
客户端感知 通常不关心代理存在 可能主动组合装饰器
示例 Spring AOP 事务代理 BufferedInputStream 包装 FileInputStream

一句话区分：

代理模式更强调“控制访问”。

装饰器模式更强调“增强功能”。

当然，在真实项目中，两者经常重叠。

例如日志代理既可以被称为代理，也可以被看作装饰器。

不要为了名字打架，重点看设计意图。

chapter 25：代理模式和适配器模式的区别

对比项 代理模式 适配器模式
目的 控制访问目标对象 转换不兼容接口
接口关系 通常和目标对象接口一致 通常接口不一致
关注点 访问控制、增强、转发 接口兼容
示例 OrderServiceProxy AliPayClientAdapter

代理模式说：

你可以访问目标对象，但要先经过我。

适配器模式说：

你的接口和我想要的不一样，我帮你转一下。

chapter 26：代理模式和外观模式的区别

对比项 代理模式 外观模式
目的 控制访问某个对象 简化复杂子系统调用
包装对象数量 通常一个目标对象 通常多个子系统
接口关系 通常和目标对象相同 通常提供更高层接口
关注点 访问控制 简化使用
示例 TransactionProxy CheckoutFacade

代理像“代办人”。

外观像“服务大厅”。

代理通常代表一个对象，外观通常封装一组子系统。

chapter 27：代理模式和责任链模式的区别

对比项 代理模式 责任链模式
结构 代理对象持有目标对象 多个处理器形成链
目的 控制对目标对象访问 请求沿链传递处理
调用目标 通常最终调用一个真实对象 可能由链上任意节点处理
示例 AOP 代理 Service FilterChain、审批链

责任链强调请求在多个处理器之间流转。

代理强调通过中间对象控制目标对象访问。

chapter 28：代理模式的优点

1. 隔离客户端和真实对象

客户端通过代理访问真实对象，真实对象可以被保护起来。

2. 可以增强目标对象

代理可以在调用前后加权限、日志、事务、监控等逻辑。

3. 符合开闭原则

不修改目标类，就能增加控制逻辑。

4. 支持延迟加载

真实对象可以等到真正使用时再创建。

5. 支持远程调用透明化

RPC 框架可以让客户端像调用本地接口一样调用远程服务。

6. 支持横切逻辑复用

动态代理可以统一处理很多类的通用逻辑。

chapter 29：代理模式的缺点

1. 增加系统复杂度

代理对象会增加一层调用结构。

2. 调试链路变长

尤其是 Spring AOP、事务、RPC、MyBatis 代理叠加时，调用链会比较复杂。

3. 动态代理有一定性能成本

反射、字节码生成、方法拦截都会有成本。

大多数业务系统中这不是瓶颈，但高性能场景要注意。

4. JDK 动态代理要求接口

没有接口时不能直接使用 JDK 动态代理。

5. CGLIB 不能代理 final 类和 final 方法

因为它基于继承和方法重写。

6. 容易产生代理失效问题

例如 Spring 中同类内部方法调用可能绕过代理，导致事务、AOP 不生效。

chapter 30：适用场景

代理模式适合以下场景。

1. 权限控制

调用目标对象前先检查权限。

2. 事务控制

方法执行前开启事务，成功提交，失败回滚。

3. 日志和监控

统一记录方法调用日志、耗时、异常。

4. 缓存代理

调用目标方法前先查缓存。

5. 远程代理

把本地接口调用转成远程网络调用。

例如 RPC、Feign、Dubbo、gRPC 客户端代理。

6. 虚拟代理

真实对象创建成本高，延迟到需要时再创建。

7. MyBatis Mapper

MyBatis 的 Mapper 接口背后也是代理对象在工作。

你没有写实现类，但可以调用接口方法。

背后就是框架生成代理对象，解析 Mapper 方法并执行 SQL。

chapter 31：不适合使用代理模式的场景

以下场景不建议使用代理模式。

1. 只是简单方法调用

如果没有访问控制或增强需求，没必要加代理。

2. 业务逻辑应该直接写在领域对象里

不要把核心业务规则藏在代理里。

代理适合横切逻辑，不适合承载核心业务。

3. 代理链太复杂

如果一个对象被代理、装饰、AOP、RPC、过滤器层层包裹，排查问题会很痛苦。

4. 对性能极端敏感且调用频率极高

高频底层方法可能不适合使用反射代理。

5. 团队不理解代理机制

Spring AOP、事务失效、动态代理类型转换等问题都需要理解代理机制。

否则出问题时很容易玄学调试。

chapter 32：真实项目中的实践建议

1. 业务代码优先面向接口

JDK 动态代理依赖接口。

即使使用 CGLIB，面向接口编程也更利于解耦。

2. 不要在代理里写核心业务

代理适合写：

  • 权限；
  • 日志；
  • 事务；
  • 缓存；
  • 限流；
  • 监控。

不适合写：

  • 创建订单核心逻辑；
  • 扣库存规则；
  • 价格计算；
  • 支付状态流转。

3. Spring 中注意自调用问题

同一个类内部方法调用不会经过代理。

如果需要事务或 AOP 生效，要通过代理对象调用，或者调整方法边界。

4. 注意 final 类和 final 方法

如果你希望 Spring AOP 或 CGLIB 代理生效，不要随意给类和方法加 final。

5. 区分代理和装饰器

如果重点是访问控制、事务、RPC，偏代理。

如果重点是自由叠加日志、重试、限流，偏装饰器。

6. 日志代理要避免打印敏感数据

比如密码、token、身份证号、手机号等。

代理层很容易统一打印参数，但别把敏感信息也统一送上日志广场。

7. 代理异常要处理好

反射调用目标方法时，异常可能被包装。

JDK 反射常见 InvocationTargetException。

真实项目中要做好异常解包和转换。

8. 不要迷信动态代理

动态代理很强，但不是所有场景都要上。

有时候一个显式的装饰器、模板方法或拦截器链更清晰。

chapter 34：一句话总结

代理模式的本质是：

不直接访问目标对象，而是通过代理对象控制对目标对象的访问，并在访问前后加入必要的控制或增强逻辑。

它特别适合：

  • 权限校验；
  • 事务控制；
  • 日志监控；
  • 缓存处理；
  • 远程调用；
  • 延迟加载；
  • 方法增强；
  • 框架拦截。

在 Java 后端开发中，代理模式非常重要。

Spring AOP、声明式事务、MyBatis Mapper、RPC 客户端、Feign、Dubbo 等技术背后都能看到代理思想。

静态代理适合理解原理。

JDK 动态代理适合基于接口的通用增强。

CGLIB 适合没有接口的类代理。

Spring AOP 则是代理模式在企业级开发中的工程化落地。

代理模式最关键的不是“会不会写 Proxy.newProxyInstance”，而是理解：

为什么要让访问经过一层代理？

如果这层代理让访问更安全、更可控、更可扩展，它就是好设计。

如果这层代理只是让调用链更长、更难查、更玄学，那它就是给代码戴了一顶没用的帽子。
