设计模式：命令模式

欢迎你来读这篇博客，这篇博客主要是关于命令模式。
其中包括命令模式的核心思想、适用场景、优缺点、命令队列、撤销操作、异步执行、与策略模式/责任链模式/事件模式的区别，以及 Java 后端开发中订单操作命令总线的完整案例。

序言

在 Java 后端开发中，我们经常会遇到这样的问题：

一个调用者想触发某个操作，但它不应该直接知道这个操作具体由谁执行、怎么执行、什么时候执行。

比如后台管理系统中有很多订单操作：

  • 创建订单；
  • 取消订单；
  • 支付订单；
  • 发货订单；
  • 关闭订单；
  • 退款订单；
  • 审核订单；
  • 批量导出订单。

最直接的写法是：

这当然可以。

但如果后面出现这些需求：

  • 操作要排队执行；
  • 操作要异步执行；
  • 操作要记录操作日志；
  • 操作失败后要重试；
  • 操作需要支持撤销；
  • 操作要统一鉴权；
  • 操作要统一落库审计；
  • 操作要支持批量组合；
  • 操作要可以延迟执行；
  • 操作要从前端配置或 MQ 消息中动态触发。

这时如果调用者直接依赖具体 Service 方法，就会越来越难扩展。

命令模式就是为了解决这类问题：

把一个请求封装成一个对象，从而让请求可以被参数化、排队、记录、撤销、重试、异步执行。

简单说：

命令模式就是把“我要做什么”封装成一个对象。

这个对象本身就是一条命令。

调用者不需要知道命令怎么执行，只需要把命令交给执行器。

这就像你在餐厅点菜，不需要冲进后厨告诉厨师“先热锅、再下油、再放菜”，你只需要下单。菜单就是命令，服务员是调用者，厨师是接收者，后厨系统是命令执行器。

当然，如果你点的是“随便”，那不是命令模式，那是产品经理需求不明确。

正文

chapter 1：什么是命令模式

命令模式，英文是 Command Pattern，属于行为型设计模式。

它的定义是：

将一个请求封装为一个对象，从而使你可以用不同的请求对客户端进行参数化，对请求排队或记录请求日志，以及支持可撤销的操作。

这句话很长，但核心只有一个：

把请求封装成对象。

在命令模式中，一个操作不再只是一个方法调用，而是一个对象。

例如：

这个 CancelOrderCommand 对象就代表“取消订单”这个请求。

它可以被：

  • 立即执行；
  • 放进队列；
  • 保存到数据库；
  • 发送到 MQ；
  • 延迟执行；
  • 批量执行；
  • 失败重试；
  • 记录日志；
  • 执行撤销。

一旦请求变成对象，它就可以像普通对象一样被传递、组合、存储和调度。

这就是命令模式最重要的价值。

chapter 2：命令模式的核心角色

命令模式一般包含五个角色：

  1. Command 抽象命令：定义命令执行接口。
  2. ConcreteCommand 具体命令：封装具体请求，调用接收者完成操作。
  3. Receiver 接收者：真正执行业务逻辑的对象。
  4. Invoker 调用者：负责触发命令执行。
  5. Client 客户端：创建命令对象，并设置接收者。

结构如下：

chapter 3：为什么需要命令模式

命令模式解决的核心问题是：

调用者和执行者之间耦合太强。

如果 Controller 直接调用具体 Service：

Controller 就必须知道：

  • 调哪个 Service；
  • 调哪个方法；
  • 传哪些参数；
  • 什么时候执行；
  • 失败如何处理；
  • 是否要重试；
  • 是否要记录操作日志。

如果操作越来越多，调用方就会越来越复杂。

命令模式把这些操作封装成对象：

调用方只负责创建命令和提交命令。

执行细节交给命令对象和命令总线。

这样可以带来几个好处：

  • 调用者和执行者解耦；
  • 操作可以被统一调度；
  • 操作可以排队；
  • 操作可以异步；
  • 操作可以记录日志；
  • 操作可以撤销；
  • 操作可以组合；
  • 操作可以延迟执行。

命令模式适合的不是普通方法调用，而是“请求需要被对象化”的场景。

例如：

3. 请求需要记录日志

命令对象可以包含操作人、操作时间、操作参数。

适合审计日志。

例如统一鉴权、幂等、重试、限流、监控。

命令总线可以在执行命令前后统一处理。

chapter 16：支持撤销的命令

命令模式有一个很经典的能力：撤销。

要支持撤销，可以定义一个新的接口。

撤销的关键是：

命令在执行前要保存足够的旧状态，撤销时才能恢复。

比如取消订单命令。

取消前订单可能是 CREATED。

执行取消后变成 CANCELLED。

撤销时要恢复成原来的状态。

chapter 35：命令模式的优点

1. 解耦调用者和执行者

调用者只依赖命令接口或命令总线，不直接依赖具体执行逻辑。

2. 请求可以对象化

一旦请求变成对象，就可以传递、保存、排队、重试、撤销。

3. 易于扩展新命令

新增命令时，新增 Command 和 Handler 即可。

4. 支持日志和审计

命令对象天然适合记录操作信息。

5. 支持撤销和重做

命令可以保存旧状态，实现 undo()。

6. 支持队列和异步

命令可以进入队列，由后台线程或 MQ 消费。

7. 支持宏命令

多个命令可以组合成一个大命令。

chapter 36：命令模式的缺点

1. 类数量增加

每个操作都可能对应一个命令类和处理器类。

2. 简单场景会显得啰嗦

如果只是一个普通方法调用，命令模式可能过度设计。

3. 命令对象设计需要谨慎

命令里放什么参数、是否可序列化、是否可重试，都需要设计。

4. 撤销并不总是容易

真实业务中的撤销往往不是简单恢复旧状态，而是需要补偿流程。

5. 异步命令会引入更多复杂度

包括幂等、重试、事务、消息丢失、顺序一致性等。

chapter 37：适用场景

命令模式适合以下场景。

1. 请求需要被参数化

例如按钮绑定不同命令。

2. 请求需要排队或异步执行

例如任务队列、后台任务、MQ 消息。

3. 请求需要日志和审计

例如后台管理操作日志。

4. 请求需要撤销和重做

例如编辑器、配置变更、工作流操作。

5. 调用者和执行者需要解耦

例如 Controller 不直接依赖复杂业务执行逻辑。

6. 需要统一命令分发

例如 CommandBus。

7. 需要把操作保存下来之后再执行

例如延迟任务、定时任务、失败重试任务。

chapter 38：不适合使用的场景

以下场景不建议使用命令模式。

1. 普通 CRUD 过于简单

如果只是：

2. 操作不需要排队、撤销、异步、日志

如果没有这些需求，直接方法调用更简单。

3. 团队不熟悉命令总线

命令总线会引入间接调用，新人可能需要理解成本。

4. 命令数量失控

如果所有操作都命令化，命令类会非常多。

需要分包、命名、规范管理。

5. 命令和事件语义混乱

不要用 OrderCreatedEvent 去表示“创建订单请求”。

也不要用 CreateOrderCommand 表示“订单已创建事实”。

语义错了，架构后面会非常别扭。

chapter 41：一句话总结

命令模式的本质是：

把请求封装成对象，让请求可以被传递、存储、排队、记录、撤销、重试和异步执行。

它特别适合：

  • 操作需要排队；
  • 操作需要异步；
  • 操作需要审计；
  • 操作需要撤销；
  • 操作需要重试；
  • 操作需要统一调度；
  • 调用者和执行者需要解耦；
  • 后端系统需要 CommandBus。

命令模式不是为了把每个方法都包装成类。

它真正适合的是那些“操作本身有生命周期”的场景。

一个普通方法调用，直接调用就好。

一个需要被记录、调度、重试、撤销、异步执行的请求，就值得被封装成命令。

好的命令模式像一张清晰的工单：谁发起、做什么、谁处理、结果如何，都有记录。

坏的命令模式像一堆小纸条：类很多，语义乱，最后没人知道哪张纸条该交给谁。
