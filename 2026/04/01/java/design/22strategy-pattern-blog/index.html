设计模式：策略模式

欢迎你来读这篇博客，这篇博客主要是关于策略模式。
其中包括策略模式的核心思想、适用场景、优缺点、与状态模式、模板方法模式、工厂模式、责任链模式的区别，以及 Java 后端开发中订单优惠折扣策略的完整案例。

序言

在 Java 后端开发中，我们经常会遇到这样一种代码：

但业务发展后，条件越来越多：

  • 普通用户折扣；
  • VIP 用户折扣；
  • SVIP 用户折扣；
  • 新用户首单折扣；
  • 满减活动折扣；
  • 会员日折扣；
  • 优惠券折扣；
  • 积分抵扣；
  • 渠道专属折扣；
  • 黑五活动折扣。

最后一个方法里堆满了 if else。

更麻烦的是，每新增一种折扣规则，就要修改原来的代码。

这会带来几个问题：

  • 方法越来越长；
  • 分支越来越多；
  • 修改旧代码容易引入 bug；
  • 单元测试越来越难；
  • 违反开闭原则；
  • 业务规则耦合在一起；
  • 新增算法不够优雅。

策略模式就是为了解决这类问题：

定义一系列算法，把每个算法封装起来，并且让它们可以相互替换。

简单说：

策略模式就是把一堆 if else 中的不同算法拆成一个个策略类。

调用方不关心具体怎么算。

调用方只需要选择一个策略，然后执行。

这就像出门有多种交通方式：

  • 步行；
  • 骑车；
  • 打车；
  • 地铁；
  • 高铁；
  • 飞机。

它们都能完成“到达目的地”这个目标，但实现方式不同。

策略模式就是把这些“不同实现方式”封装起来。

正文

chapter 1：什么是策略模式

策略模式，英文是 Strategy Pattern，属于行为型设计模式。

它的定义是：

定义一系列算法，将每一个算法封装起来，并使它们可以相互替换。策略模式让算法的变化独立于使用算法的客户端。

这个定义里有三个关键点：

  1. 一系列算法；
  2. 每个算法独立封装；
  3. 客户端可以替换算法。

在 Java 中，策略模式通常包含三个角色：

  1. Strategy 抽象策略：定义算法接口。
  2. ConcreteStrategy 具体策略：实现具体算法。
  3. Context 上下文：持有策略对象，并调用策略完成任务。

chapter 2：策略模式解决什么问题

策略模式主要解决的是：

同一类行为有多种实现方式，并且需要在运行时选择其中一种。

例如：

1. 支付方式

2. 优惠计算

3. 文件导出

4. 消息发送

5. 排序算法

6. 风控规则

这些场景都有一个共同点：

目标一致，但实现方式不同。

这就是策略模式的主场。

chapter 24：策略模式和状态模式的区别

策略模式和状态模式非常像。

它们都把行为封装到不同类中。

但意图不同。

对比项 策略模式 状态模式
核心目的 替换算法 对象行为随状态变化
是否有状态流转 通常没有 通常有
策略/状态是否稳定 策略由外部选择 状态由对象内部维护
Context 角色 使用策略 持有当前状态
示例 VIP 折扣、满减折扣 订单 CREATED、PAID、SHIPPED

一句话区分：

策略模式是“我选择哪种算法”。

状态模式是“我当前处于什么状态，所以行为不同”。

chapter 25：策略模式和模板方法模式的区别

模板方法模式是在父类中定义算法骨架，子类实现具体步骤。

策略模式是把完整算法封装成可替换对象。

对比项 策略模式 模板方法模式
复用方式 组合 继承
扩展方式 新增策略类 新增子类并重写步骤
算法结构 每个策略可以完全不同 流程骨架固定
灵活性 更灵活 流程更稳定
示例 不同折扣策略 文件导入固定流程

对比项 策略模式 责任链模式

chapter 27：策略模式和命令模式的区别

对比项 策略模式 命令模式
核心目的 替换算法 把请求封装成对象
是否保存请求参数 通常不保存具体请求生命周期 通常保存请求参数
是否支持排队/撤销 不是重点 是重点
调用方式 strategy.calculate(context) command.execute()
示例 DiscountStrategy CreateOrderCommand

策略模式关注“怎么算”。

命令模式关注“要做什么”。

优惠计算用策略。

后台操作任务、撤销、异步执行更适合命令。

chapter 28：策略模式和解释器模式的区别

策略模式适合有限算法。

解释器模式适合可配置表达式。

对比项 策略模式 解释器模式
规则形式 Java 类 表达式/DSL
扩展方式 新增策略类 新增表达式或配置规则
动态配置 不太方便 更方便
复杂度 较低 较高
示例 VipDiscountStrategy amount >= 100 AND userLevel == VIP

如果优惠规则固定几种，用策略模式。

如果优惠规则需要运营动态配置，解释器模式或规则引擎更合适。

chapter 29：策略模式的优点

1. 消除大量条件分支

把不同算法拆到不同策略类。

2. 符合开闭原则

新增策略时，尽量不修改原有业务逻辑。

3. 单一职责更清晰

每个策略只负责一种算法。

4. 可测试性更好

每个策略可以单独写单元测试。

5. 可扩展性好

新增算法只需要新增策略实现。

6. 可以运行时切换策略

根据用户类型、渠道、配置动态选择策略。

chapter 30：策略模式的缺点

1. 类数量增加

每种算法都可能对应一个策略类。

策略多了之后类会很多。

2. 调用方需要知道策略类型

调用方通常需要传入策略标识。

例如：

3. 策略选择逻辑仍然需要管理

需要工厂、路由器或 Spring 容器来选择策略。

4. 不适合算法特别少的场景

只有两个简单分支时，直接 if else 可能更清晰。

5. 策略之间如果互相依赖，会变复杂

策略应该独立。

如果策略 A 调策略 B，策略 B 又调策略 C，可能说明抽象有问题。

策略模式适合以下场景。

1. 同一类行为有多种实现

例如支付、折扣、导出、排序、消息发送。

2. 代码中有大量同类 if else

例如：

3. 算法需要灵活替换

运行时根据参数选择不同算法。

4. 算法相对独立

每个策略可以独立完成任务。

5. 新增算法比较频繁

策略模式可以降低修改旧代码的概率。

chapter 32：不适合使用的场景

以下场景不建议使用策略模式。

1. 分支很少且不会变化

两个简单判断直接写就好。

2. 策略之间差异很小

如果只是几个参数不同，可能配置化更合适。

例如折扣率不同，可以用数据库配置折扣率，而不是每个折扣率写一个策略类。

3. 策略组合顺序很重要

如果多个策略要依次执行，责任链可能更合适。

4. 规则需要动态表达

如果规则由运营配置，解释器或规则引擎可能更合适。

5. 策略选择逻辑比策略本身还复杂

如果选择策略需要大量复杂判断，可能需要重新设计业务模型。

chapter 35：一句话总结

策略模式的本质是：

将一组可替换的算法封装成独立策略类，让调用方通过统一接口使用它们，并在运行时选择具体策略。

它特别适合：

  • 支付方式选择；
  • 折扣优惠计算；
  • 文件导出格式；
  • 消息发送渠道；
  • 排序算法选择；
  • 风控规则选择；
  • 计费规则选择；
  • 不同业务类型处理。

策略模式最重要的不是“多写几个 Strategy 类”，而是：

把变化的算法从主流程中拆出去，让新增算法不再污染原来的业务代码。

好的策略模式像工具箱：每个工具各司其职，需要哪个拿哪个。

坏的策略模式像杂物间：类很多，名字很玄，最后找一个策略比直接写 if else 还痛苦。

所以使用策略模式时，记住一句话：

策略模式适合算法多且变化频繁的场景，不适合为了消灭两个 if 而把简单代码复杂化。
