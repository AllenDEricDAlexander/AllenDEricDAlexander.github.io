设计模式：适配器模式

欢迎你来读这篇博客，这篇博客主要是关于适配器模式。
其中包括适配器模式的核心思想、适用场景、类适配器、对象适配器、接口适配器，以及 Java 后端开发中第三方支付系统适配的完整案例。

序言

在软件开发中，我们经常会遇到一种很现实的问题：

我想使用一个已有功能，但它提供的接口和我当前系统需要的接口不一样。

比如你的系统内部定义了统一的支付接口：

业务层当然不希望到处写：

这会让业务代码直接依赖外部 SDK，系统会越来越乱。

适配器模式就是为了解决这种“接口不兼容”的问题。

它的核心思想是：

在旧接口和新接口之间加一层适配器，把一个类的接口转换成客户端期望的另一个接口。

生活中的例子也很直观。

你的电脑是 Type-C 接口，投影仪是 HDMI 接口，二者不能直接连接，于是你需要一个转接头。

这个转接头不会改变电脑，也不会改变投影仪，但它能让它们协同工作。

适配器模式就是代码世界里的“转接头”。

正文

chapter 1：什么是适配器模式

适配器模式，英文是 Adapter Pattern，属于结构型设计模式。

它的定义是：

将一个类的接口转换成客户端希望的另一个接口，使原本由于接口不兼容而不能一起工作的类可以协同工作。

简单说：

适配器模式就是把不兼容的接口包装成统一接口。

它通常包含三个角色：

  1. Target 目标接口：客户端期望使用的接口。
  2. Adaptee 被适配者：已经存在的类或接口，但它不符合客户端期望。
  3. Adapter 适配器：实现目标接口，并在内部调用被适配者。

结构如下：

客户端只依赖 Target，不用关心底层 Adaptee 是什么。

chapter 2：为什么需要适配器模式

适配器模式通常用于系统集成。

只要你接过第三方系统，大概率就遇到过适配器模式。

例如：

  • 接入支付宝、微信支付、Stripe；
  • 接入阿里云 OSS、腾讯云 COS、MinIO；
  • 接入短信服务商；
  • 接入物流平台；
  • 接入不同数据库驱动；
  • 接入不同消息中间件；
  • 老系统接口改造；
  • 新旧 API 兼容；
  • 第三方 SDK 封装。

这些外部系统通常都有自己的接口风格。

如果业务层直接调用它们，会出现几个问题：

  • 业务代码和第三方 SDK 强耦合；
  • 第三方 SDK 替换成本高；
  • 多个平台接口风格不统一；
  • 测试困难；
  • 异常、返回值、状态码难以统一；
  • 外部系统细节污染核心业务代码。

适配器模式的作用就是：

把外部系统的不统一，关在适配层里。

业务层看到的是统一接口。

适配层负责处理不同平台之间的差异。

chapter 3：适配器模式的核心思想

适配器模式不是为了“创造新功能”，而是为了“转换接口”。

它一般做这几件事：

  1. 参数转换；
  2. 返回值转换；
  3. 异常转换；
  4. 状态码转换；
  5. 方法名转换；
  6. 调用流程适配；
  7. 新旧系统兼容；
  8. 第三方 SDK 屏蔽。

chapter 4：适配器模式的三种常见形式

适配器模式常见有三种形式：

  1. 类适配器；
  2. 对象适配器；
  3. 接口适配器。

1. 类适配器

类适配器通过继承被适配类，并实现目标接口来完成适配。

2. 对象适配器

对象适配器通过组合被适配对象，并实现目标接口来完成适配。

3. 接口适配器

接口适配器通常用于接口方法太多，而实现类只想关注其中一部分方法的场景。

可以提供一个抽象默认适配类，给接口方法提供默认实现，子类只覆盖需要的方法。

在 Java 8 之后，接口默认方法也能承担一部分接口适配器的角色。

类适配器的优点

  • 实现简单；
  • 可以直接复用父类能力；
  • 不需要额外持有被适配对象。

类适配器的缺点

  • 依赖继承；
  • Java 不支持多继承；
  • 如果被适配类是 final，无法继承；
  • 适配器和被适配类耦合较强；
  • 不够灵活。

所以在真实 Java 项目中，类适配器用得相对少。

更常用的是对象适配器。

chapter 16：适配器里应该放什么逻辑

适配器不是业务服务，它应该主要负责“转换”和“屏蔽差异”。

适配器中适合放：

  1. 请求参数转换；
  2. 响应结果转换；
  3. 外部状态码转换；
  4. 外部异常转换；
  5. 第三方 SDK 调用；
  6. 外部字段命名映射；
  7. 外部单位转换；
  8. 协议格式转换。

例如金额单位转换：

适配器中不适合放：

  • 订单状态流转；
  • 库存扣减；
  • 优惠券核销；
  • 积分发放；
  • 复杂业务决策；
  • 事务主流程控制。

这些应该放在应用服务或领域服务中。

适配器不是垃圾桶，不要什么都往里扔。它是转接头，不是瑞士军刀。

chapter 17：适配器模式和代理模式的区别

适配器模式和代理模式都可能包装一个对象，但目的不同。

对比项 适配器模式 代理模式
目的 转换接口 控制访问或增强能力
接口是否一致 通常不一致 通常一致
关注点 兼容已有接口 增强或控制原对象
典型场景 第三方 SDK 适配 AOP、权限、缓存、远程代理
示例 把 AliPayResponse 转成 PaymentResult 给 PaymentService 加事务或日志

适配器是“接口不一样，我帮你转一下”。

代理是“接口一样，但我在前后加点东西”。

chapter 18：适配器模式和装饰器模式的区别

装饰器模式也会包装对象，但它主要用于动态增强功能。

对比项 适配器模式 装饰器模式
目的 转换接口 增强功能
接口是否一致 通常不同 通常相同
关注点 让不兼容接口能协作 给对象叠加新能力
示例 第三方短信接口适配成 SmsClient 给 SmsClient 增加限流、日志、重试

适配器解决的是“能不能用”。

装饰器解决的是“用的时候加点料”。

chapter 19：适配器模式和门面模式的区别

门面模式也常用于屏蔽复杂系统，但它和适配器模式不一样。

对比项 适配器模式 门面模式
目的 解决接口不兼容 简化复杂子系统调用
是否转换接口 是 不一定
被包装对象 通常是一个已有类或 SDK 通常是一组子系统
典型场景 旧接口适配新接口 统一封装多个复杂服务

适配器像转接头。

门面像服务大厅窗口。

一个解决接口不匹配，一个解决调用太复杂。

chapter 20：适配器模式和桥接模式的区别

适配器模式和桥接模式都能解耦抽象和实现，但使用时机不同。

对比项 适配器模式 桥接模式
使用时机 系统已经存在不兼容接口 系统设计初期主动拆分抽象和实现
目的 兼容已有类 避免多维度继承爆炸
关注点 事后补救 事前设计
示例 老系统接口适配新接口 消息类型和发送渠道分离

适配器更像“后期补丁”。

桥接更像“前期架构”。

chapter 21：适配器模式的优点

1. 解耦客户端和第三方接口

业务层只依赖系统内部统一接口，不依赖外部 SDK。

2. 提高系统扩展性

新增第三方平台时，只需要新增适配器，不需要大改业务逻辑。

3. 提高复用性

可以复用已有类或旧系统能力，而不用修改原有代码。

4. 符合开闭原则

对新增适配器开放，对已有业务代码修改较少。

5. 屏蔽外部差异

外部系统的字段、状态码、异常、协议都可以在适配层统一处理。

chapter 22：适配器模式的缺点

1. 增加类数量

每接入一个外部系统，通常就要新增一个适配器。

2. 适配层可能变复杂

如果外部接口和内部接口差异很大，适配逻辑会比较复杂。

3. 可能掩盖底层能力差异

统一接口有时候会抹平不同平台的特殊能力。

例如某支付平台支持分账，另一个平台不支持。

如果统一接口设计不好，就会很别扭。

4. 过度适配会导致抽象失真

如果为了统一而统一，把不一样的东西强行抽象成一样，系统会越来越难受。

适配器是用来降低复杂度的，不是用来制造“看起来很统一，实际很痛苦”的抽象幻觉。

chapter 23：适用场景

适配器模式适合以下场景。

1. 接入第三方 SDK

例如：

  • 支付 SDK；
  • 短信 SDK；
  • 云存储 SDK；
  • OCR SDK；
  • 地图 SDK；
  • 物流 SDK。

2. 新旧系统兼容

老系统接口不能改，但新系统希望使用统一接口。

这时可以用适配器包一层。

3. 统一多个外部服务商接口

例如多个短信服务商：

底层可以适配：

  • 阿里云短信；
  • 腾讯云短信；
  • 华为云短信；
  • 极光短信。

4. 框架集成

把某个外部组件适配成系统内部标准接口。

例如：

  • 把不同缓存客户端适配成统一 CacheClient；
  • 把不同文件存储适配成 FileStorageClient；
  • 把不同消息队列适配成 MessageProducer。

5. 历史代码改造

如果已有老代码不能轻易修改，可以写适配器让新代码以新接口调用老代码。

chapter 24：不适合使用的场景

以下场景不建议使用适配器模式。

1. 接口本来就兼容

如果两个接口已经一致，不需要额外加一层适配器。

2. 只是想偷懒隐藏坏设计

有些时候不是接口不兼容，而是抽象设计本身不合理。

这时应该重构接口，而不是不断加适配器补丁。

3. 适配逻辑比业务逻辑还复杂

如果适配器里出现大量业务判断，说明职责边界可能错了。

4. 外部能力差异过大

如果多个平台能力完全不同，强行统一接口可能会导致内部模型非常扭曲。

这种情况下可以考虑分层抽象，或者只抽象共同能力，把特殊能力单独暴露。

chapter 27：一句话总结

适配器模式的本质是：

把一个已有类或外部系统的接口转换成当前系统期望的接口，让不兼容的接口能够协同工作。

它特别适合第三方 SDK 接入、新旧系统兼容、多平台统一封装等场景。

在 Java 后端开发中，适配器模式最常见的价值是：

  • 隔离第三方 SDK；
  • 统一请求响应模型；
  • 屏蔽外部接口差异；
  • 降低业务层耦合；
  • 提高系统可扩展性。

如果说工厂模式解决“对象怎么来”，建造者模式解决“复杂对象怎么搭”，原型模式解决“对象怎么复制”，那么适配器模式解决的就是：

你说你的，我说我的，中间找个人翻译一下。

这个“翻译官”，就是适配器。
