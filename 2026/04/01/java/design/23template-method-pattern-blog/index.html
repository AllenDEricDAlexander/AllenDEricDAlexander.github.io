设计模式：模板方法模式

欢迎你来读这篇博客，这篇博客主要是关于模板方法模式。
其中包括模板方法模式的核心思想、适用场景、钩子方法、优缺点、与策略模式/责任链模式/工厂方法模式的区别，以及 Java 后端开发中数据导入流程的完整案例。

序言

在 Java 后端开发中，我们经常会遇到这种情况：

多个业务流程整体步骤差不多，但其中某几个步骤的具体实现不一样。

比如数据导入。

无论是导入用户数据、订单数据，还是商品数据，整体流程可能都差不多：

但不同导入任务的细节不同：

  • 用户导入解析的是用户字段；
  • 订单导入解析的是订单字段；
  • 商品导入解析的是商品字段；
  • 每种数据校验规则不同；
  • 每种数据保存逻辑不同。

如果每个导入类都完整写一遍流程，代码会大量重复。

如果把所有导入逻辑都塞进一个类，再用 if else 判断导入类型，代码也会越来越乱。

模板方法模式就是为了解决这类问题：

在父类中定义算法骨架，把某些具体步骤延迟到子类实现。

简单说：

模板方法模式就是父类定流程，子类填细节。

它特别适合那些“流程固定，但步骤可变”的场景。

这有点像做饭：

这个流程很稳定。

但是做红烧肉和做番茄炒蛋，具体食材和烹饪方式不一样。

模板方法模式就是把“做饭流程”固定下来，让具体菜品自己决定某些步骤怎么做。

当然，别把流程定得太死。模板方法用不好，就像领导给了你一个“创新模板”：每一步都写死，还要求你自由发挥。

正文

chapter 1：什么是模板方法模式

模板方法模式，英文是 Template Method Pattern，属于行为型设计模式。

它的定义是：

在一个方法中定义一个算法的骨架，而将一些步骤延迟到子类中。模板方法使得子类可以在不改变算法结构的情况下，重新定义算法中的某些步骤。

这个定义里有几个关键词：

  1. 算法骨架；
  2. 步骤延迟到子类；
  3. 不改变流程结构；
  4. 子类重写部分步骤。

在 Java 中，模板方法模式通常通过抽象类实现。

chapter 2：模板方法模式解决什么问题

模板方法模式主要解决的是：

多个类有相同的流程骨架，但部分步骤实现不同。

例如：

1. 数据导入流程

不同导入类型解析和保存逻辑不同。

2. 支付回调处理流程

支付宝、微信、Stripe 的验签和参数不同，但流程相似。

3. 文件导出流程

Excel、CSV、PDF 的生成方式不同。

4. 任务执行流程

不同任务执行逻辑不同。

5. 消息消费流程

不同消息业务处理不同。

这些场景的共同点是：

流程稳定，步骤变化。

这就是模板方法模式的主场。

chapter 5：模板方法为什么通常用 final

模板方法通常建议用 final 修饰：

原因是：

模板方法定义的是流程骨架，不希望子类随意改流程。

如果子类可以重写模板方法，就可能破坏父类定义的流程。

例如：

原本应该先校验再保存，结果子类改成先保存再校验。

这就很危险。

所以模板方法一般是：

子类只能重写父类允许变化的步骤。

这是一种“受控扩展”。

父类像导演，子类像演员。

演员可以演不同角色，但不能擅自把第三幕演到第一幕前面。

chapter 17：模板方法模式中的抽象步骤和具体步骤

模板方法中的步骤一般有三类。

1. 抽象步骤

父类只定义方法，子类必须实现。

适合每个子类都不同的逻辑。

2. 具体步骤

父类提供默认实现，子类通常不改。

适合通用逻辑。

3. 钩子步骤

父类提供默认空实现或默认返回值，子类可选重写。

这三类步骤组合起来，就能形成一个稳定但可扩展的流程。

chapter 18：模板方法模式和继承

模板方法模式基于继承。

这是它的优点，也是它的缺点。

优点是：

  • 父类可以复用通用逻辑；
  • 子类可以重写变化步骤；
  • 流程控制很清晰。

缺点是：

  • 继承层级可能变深；
  • 子类和父类耦合较强；
  • 父类修改可能影响所有子类；
  • Java 单继承限制扩展；
  • 子类容易依赖父类内部细节。

所以使用模板方法模式时，要注意父类设计。

父类不要太复杂。

模板方法不要太多。

否则抽象类会变成一个“业务祖宗类”，所有子类都要看它脸色。

chapter 22：模板方法模式和策略模式的区别

模板方法模式和策略模式都能复用算法，但方式不同。

对比项 模板方法模式 策略模式
复用方式 继承 组合
核心目的 固定流程骨架，子类实现步骤 封装可替换算法
流程是否固定 固定 不一定
扩展方式 子类重写步骤 替换策略对象
灵活性 较低，但流程稳定 较高
示例 数据导入流程 折扣计算策略

一句话区分：

模板方法模式适合“流程固定，步骤有变化”。

策略模式适合“算法整体可替换”。

例如：

流程固定，适合模板方法。

算法可替换，适合策略模式。

chapter 23：模板方法模式和责任链模式的区别

对比项 模板方法模式 责任链模式
流程结构 父类固定流程 多个处理器组成链
扩展方式 子类重写步骤 新增 Handler
顺序控制 父类控制 链路配置控制
是否动态调整 不方便 较方便
示例 文件导入模板 下单校验链

模板方法的流程通常固定在代码中。

责任链的处理器可以更灵活地增删和排序。

如果流程稳定，用模板方法。

如果步骤经常增删和调整顺序，用责任链。

chapter 27：模板方法模式的优点

1. 复用通用流程

公共流程写在父类中，避免重复代码。

2. 固定算法骨架

父类控制流程顺序，子类只负责变化部分。

3. 符合开闭原则的一部分

新增具体实现时，可以新增子类，不修改父类流程。

4. 便于统一处理横切逻辑

比如日志、异常、耗时统计、资源释放。

5. 子类扩展点清晰

抽象方法和钩子方法明确告诉子类哪里可以扩展。

6. 适合框架设计

框架定义流程，用户扩展部分步骤。

chapter 28：模板方法模式的缺点

1. 基于继承，耦合较强

子类依赖父类流程。

父类修改可能影响所有子类。

2. 类数量增加

每种具体流程都要一个子类。

3. 流程不够灵活

流程顺序写死在父类中，不适合经常变化的流程。

4. 子类可能被迫实现不需要的方法

如果抽象步骤设计不好，子类会写很多空实现。

5. 父类容易变复杂

抽象模板如果塞太多逻辑，会成为难维护的大类。

chapter 29：适用场景

模板方法模式适合以下场景。

1. 多个类有相同流程

例如导入、导出、回调处理、任务执行。

2. 流程固定，但部分步骤变化

这是最典型的场景。

3. 希望统一控制流程顺序

例如必须先校验，再处理，再保存。

4. 需要统一处理日志、异常、资源释放

这些都可以放在父类模板方法中。

5. 框架扩展点设计

框架定义流程，业务方实现某些方法。

chapter 30：不适合使用的场景

以下场景不建议使用模板方法模式。

1. 流程经常变化

如果步骤顺序经常调整，责任链或流程编排更合适。

2. 各实现之间差异很大

如果只是表面流程相似，实际差异巨大，强行模板会很别扭。

3. 只想替换一个算法

策略模式可能更合适。

4. 不想使用继承

如果团队更倾向组合，可以使用策略、回调或函数式接口。

5. 父类会越来越胖

如果模板父类已经很大，就要谨慎继续加逻辑。

chapter 31：真实项目中的实践建议

1. 模板方法用 final 修饰

避免子类破坏流程骨架。

2. 抽象步骤不要太多

抽象方法太多，子类实现成本会很高。

3. 钩子方法要有明确语义

例如：

不要定义太多含义模糊的钩子。

4. 父类控制流程，子类只处理变化点

不要让子类知道太多父类内部细节。

5. 通用异常处理可以放父类

例如导入失败统一记录日志。

6. 资源释放放 finally

例如文件流、数据库连接、临时文件清理。

7. 如果流程步骤需要动态增删，考虑责任链

不要硬把动态流程塞进模板方法。

8. 如果只是算法替换，考虑策略模式

模板方法不是万能替代 if else。

9. Spring 项目中可以结合 Bean 路由

多种模板实现可以注册成 Bean，然后通过 type 路由。

10. 不要让模板父类变成祖传基类

如果所有业务都继承一个超级模板类，后期维护会非常痛苦。

抽象类不是垃圾场，别什么公共代码都往里面倒。

chapter 33：一句话总结

模板方法模式的本质是：

在父类中固定算法流程，把变化的步骤交给子类实现，让子类在不改变流程结构的前提下扩展具体行为。

它特别适合：

  • 数据导入；
  • 文件导出；
  • 支付回调处理；
  • 消息消费流程；
  • 任务执行框架；
  • 资源处理流程；
  • 框架扩展点设计。

模板方法模式最重要的不是“写一个抽象类”，而是判断：

这个流程是否稳定，变化点是否清晰。

如果流程稳定，步骤变化，模板方法很合适。

如果流程经常变，步骤要动态组合，责任链可能更合适。

如果只是算法替换，策略模式可能更合适。

好的模板方法像一份清晰 SOP：流程固定，岗位职责明确。

坏的模板方法像祖传规章制度：每一步都写死，谁改谁背锅。

所以使用模板方法模式时，要记住一句话：

父类定骨架，子类填细节；流程要稳定，扩展点要克制。
