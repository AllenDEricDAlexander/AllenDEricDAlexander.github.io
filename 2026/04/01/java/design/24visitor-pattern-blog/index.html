设计模式：访问者模式

欢迎你来读这篇博客，这篇博客主要是关于访问者模式。
其中包括访问者模式的核心思想、适用场景、双分派、优缺点、与策略模式/观察者模式/组合模式/迭代器模式的区别，以及 Java 后端开发中订单费用结构访问器的完整案例。

序言

访问者模式是 23 种设计模式里比较“劝退”的一个。

很多人第一次看到它，会觉得：

明明一个方法能解决的事，为什么要绕一圈 accept(visitor) 和 visit(element)？

这个反应很正常。

访问者模式确实不适合简单业务。

它真正适合的场景是：

对象结构比较稳定，但需要在这些对象结构上不断增加新的操作。

比如：

  • AST 语法树；
  • 文件目录树；
  • 组织架构树；
  • 报表元素结构；
  • 规则表达式树；
  • 订单费用明细结构；
  • 工作流节点结构；
  • 复杂表单组件树。

这些结构里的元素类型通常比较固定。

例如订单费用明细可能有：

  • 商品明细；
  • 优惠券抵扣；
  • 运费；
  • 税费；
  • 积分抵扣。

但针对这些元素的操作可能越来越多：

  • 计算总金额；
  • 打印明细；
  • 风控检查；
  • 生成报表；
  • 导出 JSON；
  • 生成审计日志；
  • 生成财务凭证；
  • 生成页面展示 VO。

如果把这些操作都写进元素类里，元素类会越来越胖。

访问者模式的想法是：

不把所有操作塞进元素类，而是把操作封装成访问者对象，让访问者去访问这些元素并执行对应逻辑。

简单说：

访问者模式就是把“对一组对象的操作”从对象本身拆出去。

元素负责保存结构。

访问者负责执行操作。

这就像博物馆里的展品很稳定，但参观的人不同：

  • 游客看热闹；
  • 专家看年代；
  • 保安看安全；
  • 摄影师看光线；
  • 馆长看价值。

展品没变，访问者不同，得到的结果不同。

正文

chapter 1：什么是访问者模式

访问者模式，英文是 Visitor Pattern，属于行为型设计模式。

它的定义是：

表示一个作用于某对象结构中的各元素的操作。它使你可以在不改变各元素类的前提下定义作用于这些元素的新操作。

这句话有两个关键点：

  1. 对象结构中的元素比较稳定；
  2. 需要在不修改元素类的情况下新增操作。

访问者模式一般包含几个角色：

  1. Visitor 抽象访问者：声明访问不同元素的方法。
  2. ConcreteVisitor 具体访问者：实现对不同元素的具体操作。
  3. Element 抽象元素：声明 accept(Visitor visitor) 方法。
  4. ConcreteElement 具体元素：实现 accept()，通常调用 visitor.visit(this)。
  5. ObjectStructure 对象结构：维护一组元素，并允许访问者访问这些元素。

费用项是元素。

chapter 30：访问者模式的优点

1. 新增操作容易

新增一个操作时，只需要新增访问者类。

例如：

2. 操作逻辑集中

同一个操作针对不同元素的逻辑集中在一个访问者里。

3. 元素类更干净

元素类不需要塞入大量不属于自己的操作。

4. 适合复杂对象结构

特别适合 AST、树结构、组合结构。

5. 符合开闭原则的一部分

对新增操作开放，对元素类修改较少。

chapter 31：访问者模式的缺点

1. 新增元素类型困难

新增一个元素类型，需要修改访问者接口和所有具体访问者。

这是最大缺点。

2. 访问者依赖具体元素类型

访问者要知道具体元素类，耦合较强。

3. 类数量增加

每个操作一个访问者，访问者类会变多。

4. 理解成本较高

accept(visitor) 和 visitor.visit(this) 对初学者不直观。

5. 可能破坏封装

元素需要暴露足够数据给访问者。

chapter 32：适用场景

访问者模式适合以下场景。

1. 对象结构稳定

元素类型不经常变化。

例如 AST 节点类型、费用项类型、文件节点类型。

2. 操作经常变化

经常新增：

  • 打印；
  • 导出；
  • 校验；
  • 统计；
  • 转换；
  • 生成报告。

3. 需要对不同类型元素执行不同逻辑

访问者可以为每种元素提供不同 visit() 方法。

4. 不希望把操作塞进元素类

元素类保持数据结构职责。

操作类独立扩展。

5. 复杂树结构或组合结构

访问者特别适合 AST 和组合模式对象结构。

chapter 33：不适合使用的场景

以下场景不适合使用访问者模式。

1. 元素类型经常变化

如果经常新增元素类型，访问者模式会很痛苦。

2. 操作很少

如果只有一两个操作，直接写方法即可。

3. 对象结构很简单

简单 DTO 没必要访问者模式。

4. 元素不愿意暴露内部数据

访问者需要通过元素接口获取数据。

如果这会破坏封装，就要谨慎。

5. 团队不熟悉双分派

访问者模式理解成本不低。

如果团队维护困难，不要硬上。

chapter 36：一句话总结

访问者模式的本质是：

把作用于一组稳定对象结构上的操作封装成访问者，从而在不修改元素类的前提下新增操作。

它特别适合：

  • AST 语法树；
  • 文件系统树；
  • 组织架构树；
  • 订单费用结构；
  • 报表元素结构；
  • 规则表达式树；
  • 组合模式对象结构；
  • 元素稳定但操作经常增加的场景。

访问者模式最关键的判断是：

元素类型是否稳定，操作是否经常变化。

如果是，访问者模式很合适。

如果元素类型经常新增，访问者模式会让你每新增一个元素就改一堆访问者，体验会很“热闹”。

好的访问者模式像专业检查员：同一批对象，财务来算钱，风控来看风险，报表来导数据，各干各的。

坏的访问者模式像强行会诊：明明一个普通对象，非要叫十个 Visitor 排队访问，代码还没复杂，心先累了。

所以使用访问者模式时，要记住一句话：

访问者模式适合稳定结构上的多种操作，不适合普通业务对象的日常 CRUD。
