Git 速成指南：从入门到撤销、分支协作与故障恢复

欢迎你来读这篇博客。本文是一份面向日常开发的 Git 速成指南与命令速查表，覆盖仓库初始化、提交、分支、合并、变基、远程协作、撤销误操作、文件忽略、历史恢复和常见工程实践。

阅读完本文，你应该能够快速判断：

  • 文件现在位于工作区、暂存区，还是已经提交；
  • 修改错了、提交错了、推送错了，分别该用什么命令；
  • reset、restore、revert、checkout 到底有什么区别；
  • 如何正确使用分支、合并、变基和远程仓库；
  • 如何把已经被 Git 跟踪的文件移出索引，但保留本地文件。

序言

Git 最难的地方通常不是命令多，而是同一个“撤销”动作，在不同阶段需要使用完全不同的命令。

先记住 Git 的四个核心区域：

  1. 工作区（Working Tree）：你正在编辑的文件。
  2. 暂存区（Index / Staging Area）：下一次提交准备包含的内容。
  3. 本地仓库（Local Repository）：已经通过 commit 保存的历史。
  4. 远程仓库（Remote Repository）：GitHub、GitLab、Gitee 等服务器上的仓库。
flowchart LR
    A[工作区<br/>Working Tree] -->|git add| B[暂存区<br/>Index]
    B -->|git commit| C[本地仓库<br/>Local Repository]
    C -->|git push| D[远程仓库<br/>Remote Repository]

    B -->|git restore --staged| A
    C -->|git reset| B
    D -->|git fetch / git pull| C

Git 操作的本质，就是让文件和提交在这些区域之间移动。

13. reset、restore、revert 区别

命令 主要作用 是否重写历史 典型场景
git restore 恢复文件、取消暂存 否 撤销工作区或暂存区修改
git reset 移动分支指针、重置索引和工作区 是 撤销本地未公开提交
git revert 创建反向提交 否 安全撤销已推送提交
git checkout 历史命令，兼顾切分支和恢复文件 视操作而定 兼容旧教程
git switch 创建或切换分支 否 分支切换

简单记忆：

  • 文件改坏了：restore
  • 本地提交错了：reset
  • 公共提交错了：revert
  • 分支切换：switch

   merge release/1.2.0 tag: "v1.2.0"

   merge main

小型团队也可以采用更简单的 GitHub Flow：

  1. main 始终保持可发布；
  2. 每个任务创建独立功能分支；
  3. 开发完成后推送并创建 Pull Request；
  4. 代码审查和 CI 通过后合并；
  5. 合并后删除功能分支。

32. Git 故障排查思路

遇到问题时，不要第一反应就 reset --hard。推荐按以下顺序判断：

flowchart TD
    A[Git 出现问题] --> B[执行 git status]
    B --> C[确认当前分支]
    C --> D[查看 git diff]
    D --> E[查看 git diff --staged]
    E --> F[查看 git log --oneline --graph --all]
    F --> G{提交是否丢失?}
    G -->|是| H[查看 git reflog]
    G -->|否| I{是否正在 merge/rebase/cherry-pick?}
    I -->|是| J[continue / abort]
    I -->|否| K[根据工作区、暂存区、提交历史选择 restore/reset/revert]

启示录

Git 的核心不是“背命令”，而是理解三个问题：

  1. 当前修改在哪个区域？
  2. 我想保留什么，丢弃什么？
  3. 这段历史是否已经被其他人使用？

只要先回答这三个问题，大多数 Git 操作都能正确选择：

  • 工作区问题用 restore；
  • 暂存区问题用 restore --staged；
  • 本地提交问题用 reset；
  • 公共历史问题用 revert；
  • 提交找不到时用 reflog；
  • 想保留本地文件但停止 Git 跟踪时用 git rm --cached。
